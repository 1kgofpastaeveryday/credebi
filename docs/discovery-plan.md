## 4. Experiment Details

### E3: freee勤怠 vs 実給与の突合 ✅ 検証完了 (2026-03-11)

```
結果:
  12月 gross: 概算 ¥55,733 = 実際 ¥55,733 → 誤差 0% ✅
  12月 net:   実際 ¥57,003 (年末調整還付 -¥1,270 で gross より多い)
  1月 gross:  概算 ¥164,728 ≈ 実際 ¥164,715 → 誤差 ¥13 (0.008%) ✅
  1月 net:    実際 ¥164,015 (控除: 所得税 ¥700 のみ)

H3 成功基準 (±10%): 大幅クリア
  勤務給 (勤怠時間 × 時給): gross誤差 ±0.01% — 極めて高精度

発見した設計要件:
  1. 時給は期間付き履歴が必要 (effective_from/to)
     - freee側も日次ベースで期間別レート適用 (明細で確認済み)
     - 1月: 1/1-19は¥1,600, 1/20-31は¥1,800 の混在月で検証
  2. 交通費は出社日のみ支給 (リモート日は除外)
     - freee勤怠の出社/リモートタグに依存
     - タグ付けが遅延するケースあり → 交通費予測は別レイヤーで扱う
  3. 控除は学生バイトの場合ほぼゼロ (観測値)
     - 社保なし、雇用保険なし (観測月)
     - 所得税: 甲欄で月¥700程度 (1月実績)
     - 年末調整還付は予測不能な年次イベント
  4. freee API の必須パラメータ
     - company_id がクエリパラメータとして必須 (前回の /me 系は不要だった)
     - 日次レコード: /work_records/{date} で個別取得 (一括取得は /time_clocks)
     - normal_work_mins は所定労働時間 (全日480) で実勤務ではない
     - 実出勤判定は clock_in_at の有無で行う

結論: 勤務給の算出精度は十分。交通費・控除は別レイヤーとして扱い、
      confidence/provenanceで不確実性を明示する設計が妥当。
```

### E7: スプレッドシート版ドッグフード (最重要)

```
目的: アプリを作る前に「可視化で行動が変わるか」を体感検証
方法: Google Sheets に毎日5分で記録
  - 銀行残高, カード利用, 引き落とし予定, バイト見込み, 今日あと使える額
期間: 2週間
振り返り: 毎日見たか？ 行動が変わったか？ なくなったら困るか？

成功: 10/14日確認 + 行動変容2回以上
失敗パターン:
  - 3日で見なくなる → ペイン自体が弱い
  - 見るが行動変わらない → 可視化だけでは不十分
  - 手間で嫌になる → 自動化の価値は高い (ビルドすべき)
```

### E1: 友人インタビュー 5人

```
対象: クレカ持ち大学生/専門学生 5人
方法: 15分 1on1 (Mom Test準拠、誘導しない)
質問:
  1. クレカ使い始めたきっかけ
  2. 先月いくら使ったか分かるか
  3. 引き落とし日を意識しているか
  4. 残高不足でヒヤッとした経験
  5. バイト代のブレ幅
  6. 今の管理方法
  7. [聞かない: 「こういうアプリがあったら使う？」]

成功: 4/5が不透明感 + 1人以上が「怖くてカード使えない」等の回避行動
```

### E4: Google フォーム 50人

```
配布: X + 大学LINEグループ
質問: 5問以内、1分で回答
  Q1. クレカ持ってる？
  Q2. 先月の利用額を把握してる？
  Q3. 引き落とし残高を心配したことは？
  Q4. バイト給料を正確に予測できる？
  Q5. メインのメールサービスは？ (Gmail / iCloud / Yahoo / その他)

成功: Q3「よくある+たまにある」60%+ / Gmail利用率80%+
```

### E8: 大学ポスター A/Bテスト

```
目的: オフラインチャネルでの初期需要検証 + 最も刺さるメッセージング発見
場所: 大学セブンイレブン前の自由掲示スペース (昼ピーク時の行列が注目を集める)

バリアント:
  A: 質問型 — 「来月のカード引き落とし、足りる？」
  B: 共感+数字型 — 「バイト代 ¥83,000 のはずが手取り ¥71,000 だった」
  C: ソリューション+学生製型 — 「同じ大学の学生が作った、バイト×クレカの残高予測アプリ」

測定方法:
  - 各ポスターに個別QRコード (UTMパラメータ: ?utm_source=poster&utm_medium=qr&utm_campaign=poster-a/b/c)
  - 全QRコードは共通LP (ウェイトリスト登録 + E4 Googleフォーム埋め込み)
  - E1インタビューへの参加募集リンクも設置

計測指標:
  - バリアント別QRスキャン数 (LP PV)
  - LP → フォーム回答率
  - LP → ウェイトリスト登録率
  - E1インタビュー応募数

成功基準:
  - 最低1バリアントでQRスキャン率 5%以上 (推定通行量ベース)
  - LP → ウェイトリスト登録率 10%以上
  - 勝者バリアントのスキャン数が最低バリアントの1.5倍以上 (有意差)

副次効果:
  - E4 (Googleフォーム) のオフライン配布チャネルとして機能
  - E1 (インタビュー) の候補者リクルーティングにも活用
  - H1 (ペインの存在) とH5 (転換意欲) を同時検証

失敗時:
  - 全バリアント低スキャン → メッセージング以前にチャネルが不適切 or ペインが弱い
  - スキャン多いがウェイトリスト少ない → LP/価値提案の改善余地
  - バリアント間に差がない → どの切り口でも同程度に響く (良いニュース)
```

## 6. Decision Framework

### Week 3 の意思決定ポイント

```
E3 成功 + E1 成功 + E7 成功:
  → フルビルド開始 (Phase 0 → Phase 2.5 まで)
  → 最優先: メール検知 + freee連携 + ProjectionView

E3 成功 + E1 成功 + E7 失敗 (見なくなった):
  → ペインはあるが、可視化だけでは行動変容に不十分
  → 介入設計の追加検討 (Push通知の頻度・タイミング最適化)
  → スコープを「アラート特化」に絞る可能性

E3 成功 + E1 失敗 (ペイン弱い):
  → ターゲットセグメントを再検討
  → 学生ではなく若手社会人 or フリーランスにピボット？

E3 失敗 (誤差 > 15%):
  → 給与見込み算出ロジックの見直し
  → 控除テーブル精緻化で解決可能ならビルド継続

E1 + E4 で Gmail 利用率 < 60%:
  → Outlook / Yahoo 対応の優先度を上げる
  → Phase 2 のスコープ変更

E8 ポスター結果の活用:
  スキャン率高 + ウェイトリスト登録多:
    → オフラインチャネルが有効、勝者バリアントのメッセージングをLP/ストアに採用
    → ウェイトリストをベータテスターとして活用
  バリアント間に明確な差:
    → 勝者の切り口 (質問/共感/ソリューション) をブランドメッセージの軸に
  全バリアント低スキャン:
    → オフラインチャネルの限界、オンライン (X/LINE) に集中
    → ペイン自体の再検証が必要 (E1結果と合わせて判断)
```
